# 预测市场 L2 架构：三分设计

> 本文档记录「事件账本在主网、金库与预测市场在 L2」的架构选择，与原本「共识事件+博弈金库一体、预测市场」的区分，以及主网只做共识、金库与市场同链的设计理由。

---

## 1. 三分架构（与原本的区分）

### 1.1 原本的划分

- **共识事件 + 博弈金库一体**：事件创建、质押、挑战、终局与金库（存款、投票、分红）在同一套逻辑/链上。
- **预测市场**：交易 YES/NO，手续费进金库。

### 1.2 现在的三分

| 部分 | 部署位置 | 职责 |
|------|----------|------|
| **事件账本（OCP Event Ledger）** | **主网** | 共识：创建事件、质押、挑战、再质押期、终局；维护历史链、声誉、父子事件。不持有各事件的博弈代币。 |
| **博弈金库（Vault）** | **L2** | 每事件一个金库：存款即投票权重、接收预测市场手续费、按「先到先得」分红、锁定至结算。代币由本事件约定（每事件可不同）。 |
| **预测市场（Prediction Market）** | **L2** | 每事件一个市场：AMM 交易 YES/NO，手续费 donate 进**同链**的该事件金库。 |

要点：

- **金库就是 OCP 在该事件上的博弈层**，预测市场是应用层；二者同链（L2），不拆。
- **事件账本与「金库+市场」解耦**：账本只做共识输出（YES/NO/INVALID），不托管各事件的代币，也不收预测市场手续费。

---

## 2. 事件账本放在主网的优势

- **单一权威与最终性**：主网作为共识层，提供全生态统一的事件顺序与终局结果；多 L2、多应用可共用同一套事件账本，争议与声誉在一处结算。
- **共识操作低频、主网可承受**：创建事件、质押、挑战、终局等操作频率远低于预测市场的买卖；主网 gas 贵但次数少，适合做「基础设施」。
- **资产与共识分离**：每个 OCP 事件的抵押/代币可不同（USDC、ETH、某 L2 原生币等）；若账本在主网且要托管各事件代币，会变成「多资产跨链金库」，复杂且与桥类似。账本在主网、**不持币**，只输出共识；各事件的金库与市场在 L2，持币与分红同链完成，无需为手续费做跨链。
- **跨应用一致**：桥、预测市场、其他 OCP 应用都可引用同一主网事件账本做终局，无需在每个 L2 复制一套共识逻辑。

---

## 3. 金库与预测市场同链（L2）的好处

- **手续费实时进金库、先到先得分红**：每笔交易 donate 到同链金库，`accRewardPerShare` 按当前总本金即时更新；后存款者不参与此前 fee 的分配。**无需跨链**，语义简单、gas 可接受。
- **每个事件的代币可不同**：事件 A 用 USDC、事件 B 用 ETH，各自在 L2 有独立金库+市场即可，事件账本不涉及「多种代币的托管与跨链」。
- **避免「每笔交易都跨链」**：若金库在主网、预测市场在 L2，则每笔手续费要么跨链（成本高、类似桥），要么按批次跨链（丢失「每笔实时、先到先得」）。金库与市场同链则无此问题。

---

## 4. 主网如何参与「共识」

- 主网**只做事件账本**：创建事件、质押、挑战、终局均在主网合约完成；输出为「某 eventId 的终局结果」及可验证的状态/证明。
- L2 上的预测市场**读取主网共识**以结算：通过**状态证明 / 桥 / 预言机**获取主网事件账本的终局结果（YES/NO/INVALID），再在 L2 上执行 resolve、redeem、金库 withdraw 等。资金始终在 L2，不需要把手续费或金库资产跨到主网。

---

## 5. 小结

| 维度 | 设计选择 |
|------|----------|
| 事件账本 | 主网；只做共识，不持各事件代币 |
| 博弈金库 | L2；与预测市场同链，接收手续费、先到先得分红 |
| 预测市场 | L2；手续费 donate 到同链金库 |
| 共识到 L2 | 主网终局结果经状态证明/桥/预言机供 L2 解析与结算 |

这样既保留「每笔手续费实时进金库、先到先得」的清晰语义，又避免为手续费和多种代币做跨链；主网只做共识，金库与预测市场在 L2 一体部署。
