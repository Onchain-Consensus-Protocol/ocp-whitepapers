# OCP 桥应用设计

> 本文档描述基于 OCP 的跨链桥应用设计，与白皮书分离，供实现与扩展参考。核心机制见 [OCP 白皮书](./OCP_WHITEPAPER_CN.md)、[POC 白皮书](./POC_WHITEPAPER_CN.md)。

---

## 1. 定位与目标

- **问题**：传统桥依赖多签/验证者，存在单点信任与操纵风险；跨链主张需要「事实共识」而非单一数据源。
- **方案**：用 OCP 的质押—挑战—终局为**每笔跨链主张**做共识；桥秒放款（经济预最终性），事后若有争议则通过 OCP 裁决，错方被罚、用户与监督方分得桥的押金。
- **本产品为专门定制**：**不考虑声誉累加**，仅按质押金额计票，桥与监督方从零开始、规则简单。

---

## 2. 基本流程

### 2.1 参与方

| 角色 | 说明 |
|------|------|
| **用户** | 要跨链的金额（如 1w）锁入 OCP 事件金库，作为本方质押；跨链成功后拿回本金，若桥错则拿回本金 + 桥押金的一部分。 |
| **桥** | 创建 OCP 事件，默认押 YES（「我会/已按规矩在目标链放款」）；另存一笔押金入金库，**押金总量需大于用户本金**（如用户 1w、桥 2w），以保证默认 YES 票多。桥在链上秒放款，不等最终确认即可让用户在目标链提款。 |
| **巡逻机器人 / 赏金猎人** | 桥提供**公开接口**给社区，社区或赏金猎人自建机器人监督桥在 L1/L2 的跨链情况；若发现用户未收到或收少，机器人押 NO，帮用户方赢；NO 赢时桥的押金由**用户与机器人分**（用户拿本金 + 部分补偿，机器人拿赏金）。 |

### 2.2 事件与金库

- **命题**：本笔跨链是否「桥按规矩、足额在目标链向用户放款」。
- **金库**：用户要跨的金额（如 1w）+ 桥的押金（如 2w，具体只需 > 用户金额）。
- **默认**：桥创建事件并押 YES → 若无挑战，冷静期后 YES 赢。
- **经济预最终性**：是否秒放款看**压的钱够不够多**（桥押金 > 用户本金即可），不依赖抽象「置信度」；秒放款后若有异议，通过挑战与终局兜底。

---

## 3. 终局与分配规则

### 3.1 YES 赢（桥按规矩放款）

- **触发**：无人押 NO，或 YES 方总质押 > NO 方。
- **结果**：用户已在目标链收到款；金库中用户的 1w 给桥（抵桥在目标链的放款），桥的押金退回桥。

### 3.2 NO 赢（桥未按规矩 / 用户未收到或收少）

- **触发**：机器人（或任何人）押 NO，且 NO 方总质押 > YES 方。
- **结果**：桥错；**桥的押金由用户与机器人分**：
  - 用户：拿回本金 + 桥押金中的一部分（补偿）；
  - 机器人：拿桥押金的剩余部分（赏金）。
- 具体比例由协议或应用层参数定（如对半、或用户优先拿本金+补偿后余下给机器人）。

### 3.3 用户拿多了（桥多放）

- **定义**：桥自己的问题，多放了就多放了，不做专门追回或复杂分钱规则。
- **若仍走 OCP**：可视为桥错（NO）；金库分配上用户可不参与/少参与桥押金分配，桥押金在机器人与桥之间分（赏金 + 桥部分回收），由应用层定规则即可。

### 3.4 INVALID（无法判定）

- **触发**：以下情形无法判 YES 或 NO，协议或应用层判 INVALID：
  - **证据矛盾**：双方或多方证据冲突，无法形成一致事实；
  - **链重组**：依赖的区块/交易因链重组被推翻，原共识前提失效；
  - **超时**：挑战期结束仍无明确多数（YES/NO 均未过半数），或外部数据源/裁决超时未决。
- **结果**：事先定规则，例如两方退质押（用户拿回本金，桥拿回押金），或部分罚没进金库。

### 3.5 机器人押 NO 但 YES 赢

- **情形**：机器人误判、或桥/他人 YES 总质押更多。
- **结果**：按 YES 分配；机器人所押 NO 的质押被罚没（给桥或金库）。

## 4. 攻击与自平衡

### 4.1 有人拿一亿押 NO「薅羊毛」

- **结论**：不用防。出 1 亿只能争抢金库里用户 1w + 桥押金 2w，收益远低于成本，且会引来套利者/对赌方**群狼扑食**，市场自纠。

### 4.2 桥作恶还想赢（吞客户钱 + 大额押 YES）

- **结论**：桥在 YES 上押得越多，NO 赢时能吃的就越多，赏金猎人与机器人押 NO 的动力越强，**群狼扑食**。机制自平衡，无需额外设计。

---

## 5. 与主协议的关系

- **声誉**：本桥应用**不考虑声誉累加**，仅按质押金额计票；与主网 OCP / 事件账本的声誉体系独立，应用层选择。
- **白皮书**：桥的细节与案例不写入 OCP 总领白皮书，以此文档为专门设计说明；白皮书 4.2 中的「经济预最终性」「INVALID 断路器」等仍为通用表述，本文档为桥场景的落地约定。

## 6. 测试网与 Mock 代币

在无真实资金的情况下，可在测试网完整跑通桥 + OCP 流程，用于验证逻辑与集成。

- **测试网与 Gas**：以太坊测试网（如 Sepolia、Holesky）与 Base 测试网均提供**官方水龙头**，可领取免费测试 ETH 用于支付 gas，足够完成部署、创建事件、质押、挑战与终局。
- **Mock 代币**：金库与桥使用的「跨链资产」「押金」在测试网可用 **Mock 代币** 代替：
  - 部署一个可 `mint` 的 ERC-20 合约，在测试网为自己铸造大量测试用代币，作为「测试用稳定币」或跨链面额；
  - 或直接使用测试网原生 ETH 作为质押/计价单位（测试网 ETH 无真实价值，本质等同 mock）。
- **流程**：用户「跨 1w」= 锁 1w 单位 mock 代币；桥「押 2w」= 押 2w 单位 mock 代币；机器人押 NO 同样使用 mock。逻辑与主网一致，仅资产为测试用。
- **跨链模拟**：测试网可简化为「L1 锁 mock → 发事件/消息 → L2 合约按规则 mint 或释放 mock」；两条链各部署 mock 代币或 L2 按 L1 事件给用户发测试币，无需真实跨链流动性即可验证端到端流程与巡逻机器人逻辑。

### 6.1 有争议场景的模拟

在测试网要验证「用户没收到 / 收少 → 机器人押 NO → NO 赢 → 用户与机器人分桥押金」等争议路径，可采用：

- **故意不释放或错放**：桥的 L2 侧在测试网做成**可配置**（合约开关或环境变量），对某笔跨链选择「不释放」「少释放」或「释放到错误地址」。争议的**触发**可以人工或配置完成；真正要验证的是后续的自动响应。
- **巡逻侧必须自动执行**：**发现问题、质押挑战、与 OCP/链的信息通讯**应由合约或链下脚本自动完成，而不是长期依赖人工点选。测试网上也应部署「监督合约」或守护进程：监听 L1/L2 状态（或订阅桥的公开接口），当检测到某笔跨链未按规放款时，**自动**调用 OCP 的 stake(NO) 并传入对应 eventId 与质押量。这样测的是完整自动化路径，而不是一次性人工 stake。
- **测试流程建议**：  
  - 桥侧：用配置/人工触发一笔「不释放或少释放」。  
  - 巡逻侧：由**合约或脚本**读取 L2 状态（或桥暴露的「待检查列表」），判定该笔异常后自动发起 stake(NO)。  
  - 验证：终局为 NO，金库按 §3.2 分配；并可再测「桥正常放款、巡逻误判仍押 NO」时，终局 YES、NO 方被罚没（§3.5）。
- **最小可自动化范围**：至少将「对指定 eventId 调用 OCP.stake(NO, amount)`」封装成合约或脚本，由同一逻辑在检测到异常时触发；信息通讯（如把 eventId、证据摘要上报或写入链上）也由该流程自动完成，便于主网接同样的自动化。
